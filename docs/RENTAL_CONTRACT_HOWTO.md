# Как сгенерить договор аренды электромотоцикла

## Коротко

Пишешь в Slack: **"@codex сделай документ falcon-gt с завтра 18:00 до 28.05 10:00"** — и в Телегу прилетает готовый DOCX-договор, заполненный данными мота и арендатора.

## Как это работает

1. **Запрос** — ты пишешь фразу в Slack (или запускаешь скрипт вручную). Бот вытаскивает из фразы: какой мот и даты аренды
2. **Поиск мота** — скрипт ищет в Supabase (таблица `cars`, type=bike/ebike), подбирает лучший матч по fuzzy-score
3. **Данные арендатора** — берутся из OCR паспорта и ВУ (файлы JSON), которые бот получает заранее
4. **Заполнение шаблона** — HTML-шаблон договора заполняется ~40 переменными (mustache `{{variable}}`): VIN, марка, модель, цена, ФИО, паспорт, даты и т.д.
5. **Генерация DOCX** — заполненный HTML конвертится в DOCX через cheerio (с сохранением форматирования: таблицы, жирные заголовки, отступы)
6. **Доставка** — DOCX отправляется в Телегу через Bot API
7. **Верификация** — от сгенерированного файла считается SHA256-хеш и сохраняется в БД (подробнее ниже)

## Таблица rental_contract_artifacts

Это таблица в Supabase, куда сохраняется «слепок» каждого сгенерённого договора:

| Поле | Что хранит |
|------|-----------|
| `contract_key` | Уникальный ключ договора (формат: `rental-{bikeId}-{timestamp}`) |
| `requested_bike_id` | Что просили (как написали в фразе) |
| `resolved_bike_id` | Что нашли в базе (реальный ID мота) |
| `telegram_chat_id` | В какой чат отправили |
| `telegram_message_id` | ID сообщения в Телеге |
| `renter_full_name` | ФИО арендатора |
| `rent_start_date` | Дата начала аренды |
| `rent_end_date` | Дата окончания аренды |
| `doc_verifier_id` | Ссылка на запись в `doc_verifier_records` (антиподделка) |
| `original_sha256` | SHA256-хеш исходного DOCX-файла |
| `created_at` | Когда создали |

## Верификация (антиподделка)

Да, договор уже сохраняется с верификацией. Вот как:

1. После генерации DOCX от байтов считается **SHA256-хеш**
2. Хеш записывается в `rental_contract_artifacts.original_sha256`
3. Если включён `doc_verifier` (веб-апп пайплайн), хеш также регистрируется в таблице `doc_verifier_records` — это антиподделочная система: если кто-то изменит DOCX после подписания, хеш не совпадёт
4. При записи делается **read-after-write** проверка — скрипт перечитывает строку из БД и падает, если запись не подтвердилась

Включается флагом `--saveMetadata 1` при запуске скрипта (по умолчанию выключено, включать после того как миграция накачена).
