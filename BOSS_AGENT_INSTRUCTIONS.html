1 Role Separation
BOSS Strategic Architect

👑 Boss Role — Strategic Architect

  • Receives dream trailer + bio dump from client
  • Validates dream — is it achievable? Does it align with available stack?
  • Decomposes dream into quest chain (5–8 high-level doors)
  • Writes POLISH INSTRUCTIONS inside each SupaPlan task's metadata
  • Focuses on BIG PICTURE — the narrative arc, the emotional journey
  • Does NOT micromanage achievement names or micro-branching
  • Sets overall quest chain name, theme, and endpoint
  • Creates SupaPlan tasks — all initial doors are suggested_by: "boss"
  • Sends the first INVITE notification to player
What Boss writes vs. doesn't
✍️ Writespolish_instructions, dream_name, tags, spoil_preview, requires/unlocks graph, difficulty, quest_chain_id
❌ Doesn'tachievement names, achievement descriptions, notification Part 2 content, agent suggestions, commit bodies
🎯 Ownsstrategic decomposition, first INVITE notification, quest chain coherence, dream validation
🚫 Delegatescreative execution, on-the-fly naming, branching selection, commit structure, invoicing flow
AGENT Creative Executor

🤖 Agent Role — Creative Executor

  • Picks up SupaPlan task via cheatcode
  • Reads polish_instructions from task metadata
  • Executes the polish/implementation
  • IMAGINES achievement name + description on the fly — "What achievement did the player earn?"
  • Checks SupaPlan for related non-done tasks → picks TOP 3 from boss-created open tasks
  • Can SUGGEST a 4th door (✨ AGENT SUGGESTED) — either polishing current task OR genuinely new task
  • Creates structured commit body with achievement data
  • Sends notification Part 2 with achievement + branching choices
  • For SuperBoss: triggers create_invoice() → update_invoice_status('paid') flow
What Agent writes vs. doesn't
✍️ Writesachievement_name, achievement_description, commit body, notification Part 2, optional 4th door (SupaPlan task on the fly), invoice triggers
❌ Doesn'tquest chain decomposition, polish_instructions, dream validation, first INVITE, strategic vision
🎯 Ownscreative naming, branch ranking, code execution, commit structure, on-the-fly SupaPlan task creation
🚫 Delegatesstrategic vision, quest coherence, overall difficulty balancing, dream validation

2 SupaPlan Schema — How Tasks Store Everything
Core Fields
idUUIDThe "cheatcode" / door key
titlestringQuest name like "Архитектор Ботов"
statusenumopen claimed running ready_for_pr done
bodytextTask description for Codex
metadataJSONBAll quest chain data lives here ↓
Metadata JSONB — Full Structure
{ "quest_chain_id": "string" // links doors together "chain_position": "number" // 1/6, 2/6, etc. "polish_instructions": "string" // boss-written creative direction "codex_task_url": "string" // direct link to Codex task "player": "string" // who this door is for "dream_name": "string" // the overall dream/wish name "difficulty": "easy|medium|epic" "tags": ["string"] // for dynamic branching "suggested_by": "boss|agent" // who proposed this door "requires": ["UUID"] // doors that must complete first "unlocks": ["UUID"] // doors this enables after completion "spoil_preview": "string" // spoiler text for notification }
Dependency Graph — requires/unlocks
The requires and unlocks arrays form a DAG (directed acyclic graph) of door dependencies.
Door 1 root
requires: []  →  unlocks: [D2, D3]
↓ ┌───────────────┐ ↓
Door 2a red
requires: [D1]
unlocks: [D4]
Door 2b blue
requires: [D1]
unlocks: [D5]
↓
Door 3 merge
requires: [D2 OR D3]  →  unlocks: [D6]
Status Lifecycle
⬤ open — created by Boss, waiting for player
↓ cheatcode entered
⬤ claimed — player entered cheatcode, agent assigned
↓ agent starts execution
⬤ running — agent executing polish instructions
↓ agent finishes
⬤ ready_for_pr — code ready, PR can be created
↓ player hits PR button or continues
⬤ done — complete, achievement locked in

3 Branching Protocol — How Agents Pick Next Doors
STEP 1: Query SupaPlan
When agent completes a task and prepares Notification Part 2:
CHECK SupaPlan for tasks where: status != 'done' quest_chain_id == current chain requires[] includes current task UUID (or requires is empty) suggested_by == 'boss' // TOP 3 from boss-created tasks only NOT the current task itself
STEP 2: Rank by Relevance
★ Highest priority — tasks in current task's unlocks[]
◆ Medium priority — tasks with overlapping tags[]
● Low priority — remaining chain tasks
STEP 3: Pick TOP 3 (from boss-created open tasks)
Select the top 3 ranked tasks from boss-created open tasks only. These become the 🔴🔵🟢 door options.
🔴 Door Xa
Highest rank
🔵 Door Xb
Medium rank
🟢 Door Xc
Lowest of top 3
STEP 4: Optional 4th — Agent Suggested
The 4th option can be one of TWO types:
TYPE A Polish current task
Additional minor enhancement in the same cheatcode area. Quick touch-up that makes the current door even better.
TYPE B Genuinely new door
Agent CREATES a full SupaPlan task ON THE FLY. Gets suggested_by: "agent", requires: [current-UUID], quest_chain_id: same.
  • A Type A: Extra polish pass on current task — "one more thing that would make this shine"
  • B Type B: Agent creates a brand-new SupaPlan task on the fly → full door with metadata
  • → Both types appear as ✨ AGENT SUGGESTED in notification
  • → Type B tasks are persisted in SupaPlan and join the dependency graph
STEP 5: Format Notification Output
🔴 Door Xa: [name] — [one-line description] 💬 [spoil_preview from metadata] 🔵 Door Xb: [name] — [one-line description] 💬 [spoil_preview from metadata] 🟢 Door Xc: [name] — [one-line description] 💬 [spoil_preview from metadata] ✨ Door Xd: [name] — [description] (AGENT SUGGESTED) 💬 [spoil_preview]
Branching Algorithm — Visual
The complete decision pipeline from task completion to notification:
✓ Task complete — agent ready to branch
↓

↓

↓
3 Pick TOP 3 → 🔴🔵🟢 door cards
↓

📨 Format 3+1 notification → send

Output Format

13/10 tier — transcendence-level naming
Naming Rules
  • → UPPERCASE with hyphens: ARCHITECT-OF-BOTS
  • → 2–4 words max — punchy, not a sentence
  • → Must relate to what was actually built
  • → Should feel like a game achievement — earned, not assigned
  • → Description = one line, past tense, what they did
  • → spoil_preview = curiosity hook for the NEXT door (not this one)
  • → SuperBoss (13/10) achievements get an extra flourish — transcendence-tier naming
  • → Agent-suggested doors get achievements too — same quality bar

7 Agent Suggestion Pipeline — "West Coast Style"
When to Suggest
During task execution, if agent feels something is missing:
  • → The bot has commands but no /help system
  • → The API exists but there's no rate limiter
  • → The feature works but lacks error handling
  • → A logical next step isn't in the chain
  • → Current task could use one more polish pass (Type A)
Two Types of Suggestions
TYPE A Additional Polish of Current Task
Minor enhancement in the same cheatcode area. "One more thing that would make this shine."
Example: current task built /stats command → Type A suggests "add emoji reaction to /stats output"
TYPE B Genuinely New Door
Agent creates a full SupaPlan task ON THE FLY. Entirely new door in the chain.
Example: current task built auth → Type B creates "add /help auto-generator" as a new door
How to Create — On The Fly SupaPlan Task
Agent creates a real SupaPlan task in the database:
INSERT INTO supaplan_tasks (id, title, status, body, metadata) VALUES ( gen_random_uuid(), // new cheatcode 'Auto-Help Generator', // or whatever fits 'open', 'Generate /help from command registry', '{ "suggested_by": "agent", "requires": ["current-task-UUID"], "quest_chain_id": "same-chain-id", "tags": ["help", "commands", "docs"], "spoil_preview": "Your commands will finally explain themselves...", "player": "player-name", "dream_name": "same-dream", "difficulty": "easy" }'::jsonb );
Pipeline Steps
1 Agent notes gap during execution (or potential polish)
↓
2 Decide: Type A (quick polish) or Type B (new SupaPlan task)
↓
3 For Type B: INSERT INTO supaplan_tasks with suggested_by: "agent"
↓
4 Mention in Notification Part 2 as ✨ 4th option
↓
5 If player chooses it → normal cheatcode flow → same commit/achievement pipeline
West Coast Rules
ONE suggestion at a time — not a flood
Must be genuinely achievable with current stack
Must relate to what was just built (natural next step)
Suggestion quality = agent quality. No filler.
If no gap exists → no suggestion. Zero is fine.
Type A only if current task genuinely benefits from one more pass
Type B only if there's a real logical gap in the boss's decomposition

9 Career Profile Schema
Achievements Storage
Achievements are stored in two places:
Git commits = source of truth (eternal record)
Structured commit bodies enable one-liner retrieval from git history
Supabase = cache layer (for display/query)
Synced on PR merge. Powers career profile rendering.
Supabase options for achievement storage:
Option A: users.badges JSONB
// In public.users, badges column: badges: [ { "name": "NEXUS-WEAVER", "description": "Threaded the bot's command core", "score": 10, "pill": "red", "quest_chain_id": "dream-bot-empire", "door": "3/6", "earned_at": "2025-01-15T12:00:00Z", "suggested_by": "boss", "commit_hash": "abc1234" } ]
Option B: Separate achievements table
CREATE TABLE achievements ( id UUID PRIMARY KEY, user_id TEXT NOT NULL REFERENCES users(user_id), achievement_name TEXT NOT NULL, achievement_description TEXT, score INTEGER, // 9, 10, or 13 pill TEXT, // 'blue', 'red', 'superboss' quest_chain_id TEXT, door TEXT, // e.g. "3/6" commit_hash TEXT, suggested_by TEXT, // 'boss' | 'agent' tags TEXT[], created_at TIMESTAMPTZ DEFAULT now() );
Leaderboard View
CREATE OR REPLACE VIEW leaderboard AS SELECT u.user_id, u.username, u.full_name, u.avatar_url, COALESCE(SUM(a.score), 0) AS total_score, COUNT(a.id) AS achievements_count, COUNT(CASE WHEN a.pill = 'red' THEN 1 END) AS red_pills, COUNT(CASE WHEN a.pill = 'blue' THEN 1 END) AS blue_pills, COUNT(CASE WHEN a.pill = 'superboss' THEN 1 END) AS superboss, COUNT(DISTINCT a.quest_chain_id) AS chains_completed FROM public.users u LEFT JOIN achievements a ON u.user_id = a.user_id GROUP BY u.user_id, u.username, u.full_name, u.avatar_url ORDER BY total_score DESC;
Score Calculation
9 points per blue pill achievement
10 points per red pill achievement
13 points per superboss achievement
Total score = sum of all achievement scores.
Chains completed = count of distinct quest_chain_ids where all doors done.
Query Examples
-- Player's full career SELECT * FROM achievements WHERE user_id = 'telegram-123456' ORDER BY created_at DESC; -- Global leaderboard SELECT * FROM leaderboard ORDER BY total_score DESC LIMIT 50; -- Chain progress SELECT door, score, achievement_name, pill FROM achievements WHERE quest_chain_id = 'dream-bot-empire' AND user_id = 'telegram-123456' ORDER BY created_at; -- Player badges from users table (Option A) SELECT badges FROM public.users WHERE user_id = 'telegram-123456'; -- Git one-liner for achievement history git log --grep="player: alex" --format="%H %s" | head -20 -- SuperBoss hall of fame SELECT * FROM achievements WHERE pill = 'superboss' ORDER BY created_at DESC;
