# MapRiders Design Document v2.0
## Next-Generation Real-Time Tracking Platform

---

## Executive Summary

Current implementation provides solid real-time tracking foundation with Supabase, but requires architectural refinement for scale, UX polish, and maintainability. This document outlines best practices from leading map-based applications (Strava, Uber, Telegram Live Location, Google Maps) and proposes a comprehensive redesign.

---

## 1. Current State Analysis

### ✅ Strengths
- **Real-time architecture**: `live_locations` + broadcast pattern is production-ready
- **Database design**: Proper separation of live vs historical data
- **Throttling logic**: 3s + 15m distance thresholds prevent DB spam
- **Map engine abstraction**: Leaflet/VibeMap switch shows good foresight
- **Session management**: Complete ride lifecycle tracking

### ⚠️ Critical Issues
1. **UI/UX Chaos**: Floating controls, inconsistent spacing, poor mobile ergonomics
2. **State Management**: Multiple `useState` without proper orchestration
3. **Performance**: No marker clustering, potential re-render storms
4. **Accessibility**: Missing keyboard navigation, screen reader support
5. **Offline Support**: No queue/retry mechanism for dropped connections
6. **Battery Drain**: GPS watching without intelligent sampling
7. **Visual Hierarchy**: No clear primary/secondary actions
8. **Error Handling**: Silent failures, no user feedback

## 2. Industry Best Practices Research

### 2.1 Real-Time Location Apps

**Telegram Live Location**
- ✅ Auto-expire markers (1/5/15/60 min presets)
- ✅ Smooth marker interpolation between updates
- ✅ Battery-optimized background updates
- ✅ One-tap stop sharing

**Strava**
- ✅ Segment leaderboards with filters
- ✅ Activity feed with social features
- ✅ Route heatmaps (aggregated, privacy-preserving)
- ✅ Performance metrics visualization

**Uber/Lyft**
- ✅ Optimistic UI updates
- ✅ Driver/rider state machines
- ✅ ETA calculations with confidence intervals
- ✅ Geofencing for pickup/dropoff

**Google Maps Timeline**
- ✅ Activity detection (walking/driving/cycling)
- ✅ Place detection (auto-label stops)
- ✅ Privacy controls per-location
- ✅ Data export capabilities

### 2.2 Map UX Patterns

**Interaction Design**
- **Tap vs Long-Press**: Tap for info, long-press for actions
- **Bottom Sheets**: Swipeable panels for details (Google Maps pattern)
- **Floating Action Buttons**: Primary action always accessible
- **Contextual Toolbars**: Show relevant controls based on state

**Visual Design**
- **Z-index layering**: Markers > Routes > Map tiles > UI chrome
- **Color semantics**: Blue=self, Green=active, Gray=inactive, Red=alert
- **Motion design**: 300ms transitions, ease-in-out curves
- **Depth cues**: Shadows, elevation, blur effects

**Performance**
- **Marker clustering**: Combine nearby markers at low zoom
- **Viewport culling**: Only render visible elements
- **Debounced pan/zoom**: 150ms delay before re-query
- **Progressive loading**: Load high-priority data first

```
└── MapRidersProvider (Context)
    ├── MapCanvas (Leaflet wrapper)
    │   ├── RiderMarkersLayer (clustered)
    │   ├── ActiveRoutesLayer (polylines)

    ├── BottomSheet (draggable)
    │   ├── RiderDetailPanel
    │   ├── SessionControls
    │   └── MeetupCreator
    ├── TopBar (stats + filters)
    ├── FloatingActionButton (primary action)
    └── ToastNotifications
```

### 3.2 State Management Strategy

**Replace multiple useState with Zustand**

```typescript
// stores/mapRidersStore.ts
interface MapRidersState {
  // Session state
  activeSession: Session | null
  isRecording: boolean
  
  // Live data
  riders: Map<UserId, RiderPosition>
  meetups: Meetup[]
  
  // UI state
  selectedRiderId: UserId | null
  bottomSheetOpen: boolean
  mapBounds: GeoBounds
  
  // Actions
  startSession: () => Promise<void>
  stopSession: () => Promise<void>
  updatePosition: (pos: Position) => void
  createMeetup: (pos: LatLon) => Promise<void>
}
```

**Benefits:**
- Single source of truth
- DevTools integration
- Persistence middleware
- Computed selectors

### 3.3 Real-Time Data Flow

```
GPS Sensor
    ↓ (throttle: 3s, 15m)
PositionQueue (FIFO, max 100)
    ↓
Broadcast Channel (instant)
    ↓
live_locations UPSERT (every 30s)
    ↓
Supabase Realtime (postgres_changes)
    ↓
Client Store (Zustand)
    ↓
React Components (selectors)
```

**Optimization:**
- Use `useShallow` selector to prevent re-renders
- Memoize marker components with `React.memo`
- Virtualize long lists (recent rides)

## 6. Performance Optimization

### 6.1 Rendering Strategy

**Marker Virtualization**
```typescript
// Only render markers in viewport + 20% buffer
const visibleRiders = useViewportFilter(riders, mapBounds, {
  buffer: 0.2
})

// Cluster markers when zoom < 14
const clusteredMarkers = useMarkerClustering(visibleRiders, zoomLevel)
```

**Memoization**
```typescript
const RiderMarkerComponent = memo(({ rider }) => {
  // Only re-render if position changed > 5m
  const shouldUpdate = usePositionChanged(rider, 5)
  
  if (!shouldUpdate) return null
  return <Marker position={rider.position} />
}, (prev, next) => {
  return distance(prev.rider, next.rider) < 5
})
```

### 6.2 Network Optimization

**Request Batching**
```typescript
// Batch location updates (max 10 per request)
const locationQueue = useRef<Position[]>([])

const flushQueue = async () => {
  if (locationQueue.current.length === 0) return
  
  await fetch('/api/map-riders/batch-locations', {
    method: 'POST',
    body: JSON.stringify({
      sessionId,
      points: locationQueue.current.slice(-10)
    })
  })
  
  locationQueue.current = []
}

// Flush every 5s or when queue > 10
useInterval(flushQueue, 5000)
```

**Incremental Loading**
```typescript
// Load recent rides first, then paginate
const { data, fetchNextPage, hasNextPage } = useInfiniteQuery({
  queryKey: ['sessions', userId],
  queryFn: ({ pageParam = 0 }) => fetchSessions(userId, pageParam),
  getNextPageParam: (lastPage) => lastPage.nextCursor
})
```

### 6.3 Battery Optimization

**Intelligent GPS Sampling**
```typescript
const useSmartGPS = (isActive: boolean) => {
  useEffect(() => {
    if (!isActive) return
    
    let watchId: number
    
    const enableHighAccuracy = isRecording // Only high accuracy when recording
    const timeout = isRecording ? 10000 : 30000
    const maximumAge = isRecording ? 0 : 60000
    
    watchId = navigator.geolocation.watchPosition(
      handlePosition,
      handleError,
      { enableHighAccuracy, timeout, maximumAge }
    )
    
    return () => navigator.geolocation.clearWatch(watchId)
  }, [isActive, isRecording])
}
```

**Background Throttling**
```typescript
// Reduce update frequency when tab hidden
useEffect(() => {
  const handleVisibilityChange = () => {
    if (document.hidden) {
      setUpdateInterval(30000) // 30s when background
    } else {
      setUpdateInterval(3000)  // 3s when foreground
    }
  }
  
  document.addEventListener('visibilitychange', handleVisibilityChange)
  return () => document.removeEventListener('visibilitychange', handleVisibilityChange)
}, [])
```

## 8. Privacy & Security

### 8.1 Location Privacy
```typescript
// User controls
interface PrivacySettings {
  visibility: 'private' | 'crew' | 'public'
  shareExactLocation: boolean
  autoExpireMinutes: number | null
  blurHomeLocation: boolean // Fuzz coordinates within 500m
}

// Server-side enforcement
const sanitizePosition = (pos: Position, userId: string, viewerId: string) => {
  const settings = getUserPrivacySettings(userId)
  
  if (settings.visibility === 'private') return null
  if (settings.visibility === 'crew' && !isSameCrew(userId, viewerId)) return null
  
  if (settings.blurHomeLocation && isNearHome(pos, userId)) {
    return blurCoordinates(pos, 500) // Add random offset up to 500m
  }
  
  return pos
}
```

### 8.2 Data Retention
```sql
-- Auto-delete old live locations
CREATE POLICY "Auto-expire live locations"
ON live_locations
FOR DELETE
USING (updated_at < NOW() - INTERVAL '24 hours');

-- Archive completed sessions after 90 days
CREATE EXTENSION pg_cron;
SELECT cron.schedule(
  'archive-old-sessions',
  '0 3 * * *', -- 3 AM daily
  $$UPDATE map_rider_sessions SET archived = true WHERE created_at < NOW() - INTERVAL '90 days'$$
);
```

## 12. Risk Mitigation

| Risk | Impact | Probability | Mitigation |
|------|--------|-------------|------------|
| Supabase rate limits | High | Medium | Implement request queue + exponential backoff |
| GPS drift in urban areas | Medium | High | Snap-to-road algorithm, filter outliers |

| Privacy concerns | Critical | Medium | Granular controls, data retention policy |
| Map tile loading slow | Medium | Medium | Cache tiles, use CDN, lazy loading |
